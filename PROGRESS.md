# Progress Tracker — Weekly Report Generator & Team Dashboard

Local, working tracker for development. Not a spec — `AGENTS/*.md` remains authoritative.
Update this file as tasks move: check items off, add dated notes under "Log" for anything
non-obvious (decisions, deviations, blockers).

## Objective

Build the full-stack app per `AGENTS/technical_assignment.md`, in strict phase order:
**Backend → Frontend → AI (bonus)** (`AGENT_RULES.md §5`). Monorepo already scaffolded
(Turborepo, NestJS `apps/api`, Next.js `apps/web`, `packages/shared-types`), Prisma schema
and seed already written, Docker Compose stack builds successfully. Feature modules
(`auth`, `users`, `reports`, `projects`, `dashboard`, `ai`) currently exist only as empty
`@Module({})` stubs — this tracker starts from there.

- 2026-07-08 — **Bug (user-reported), fixed:** registering through the real `docker compose up` stack failed with `ECONNREFUSED 127.0.0.1:3001` inside the `web` container. Root cause: `next.config.mjs`'s `rewrites()` destination is resolved once by `next build` and frozen into `.next/routes-manifest.json` — the standalone server never re-reads it at runtime, so the `API_URL: http://api:3001` set under `environment:` in `docker-compose.yml` had zero effect; the image had always baked in the `http://localhost:3001` fallback since the `builder` stage never had that env var at build time. This bug predates this session's frontend work — it was never caught earlier because every prior verification used `npm run dev` locally (pointed at the host-mapped port), never the actual Dockerized `web` service talking to the Dockerized `api` service. Fixed by passing `API_URL` as a Docker **build arg** (`ARG`/`ENV` in `apps/web/Dockerfile`, `build.args` in `docker-compose.yml`) instead of a runtime environment variable. Verified by inspecting the rebuilt `routes-manifest.json` (now correctly shows `http://api:3001`) and registering through the real Dockerized stack end-to-end in a headless browser — the exact path the user hit.
